## 핵심 네러티브 (확정본 v2)

기반시설 규제 대상 조직의 보안 담당자 한 명이 수백 개가 넘는 컨테이너를 혼자 점검한다. 그에게 점검 기준은 **선택지가 아니다** — 감사에서 인정받으려면 KISA 「주요정보통신기반시설」 항목을 그대로 써야 한다. 그래서 진짜 고통은 '무엇을 점검할까'가 아니라, *이미 정해진 무거운 표준을 수백 개 컨테이너에 손으로 돌리는 일*, 그리고 *표준이 뱉는 raw pass/fail을 사람이 쓸 수 있는 개선안으로 옮기는 일*이다. 이 도구는 그 둘을 각각 자동화와 LLM 번역으로 없앤다. 즉 **표준은 규제가 정해주고(맥락), 도구의 승부수는 그 표준을 스케일에서 돌리고 '읽히게' 만드는 해석 품질**이다. 컨테이너 특성상 KISA 103개 중 서비스·웹 항목은 대부분 skip이지만, 그 skip 목록조차 "표준 전체를 훑었다"는 감사 근거가 되고, 실제 fail이 잡히는 계정·파일권한 축에서 Claude의 해석 품질이 도구의 가치를 증명한다.

## 구성 요소 (모두 사용자 확인 완료)

| 요소 | 내용 | 근거 |
|------|------|------|
| **대상** | 기반시설 규제 대상 조직에서, 수백 개+ 컨테이너를 혼자 점검하는 보안 담당자 | 확인 |
| **진짜 문제** | ① 강제된 표준을 수작업으로 수백 개에 못 돌림(스케일의 벽) ② raw 결과 해석·정리 부담 | 확인 |
| **맥락·타이밍** | 반복 노동이 한계치 + **KISA 준수가 규정·감사로 강제됨** + LLM 번역이 이제 가능 | 확인 |
| **핵심 관점·베팅** | 표준은 주어진 전제. 차별점은 **"강제 표준을 스케일에서 자동 실행 + 사람이 읽을 리포트로 번역"하는 해석 품질**. 커버리지는 규제가 요구하는 최소선, 승부처는 해석 | 확인 |
| **해결 방향** | 무개입 파이프라인 + KISA를 playbook으로 코드화 + LLM 해석 + 대시보드 집약 | 명시 |
| **성공의 모습** | 무개입 E2E + 취약/안전 대비 + fail 항목의 읽히는 개선안 | 명시 |

## 진단 — 이전 분열의 봉합

- **봉합된 것:** 이전 버전의 "해석 품질 vs 표준 커버리지" 분열이 위계로 정리됨. → **커버리지 = 규제가 정한 전제, 해석 = 차별점(베팅).** 두 힘이 충돌이 아니라 층을 이뤄 한 이야기로 합쳐진다.
- **왜 KISA인가 (이전엔 빠졌던 요소):** 단순 편의가 아니라 **실제 규정·감사 대상**이기 때문. 이로써 '규정 준수(compliance)'가 네러티브의 정식 맥락 축으로 승격.

## 실행 유의점 (네러티브 아님, 정합성 메모)

- KISA는 완전 서버 기준이라 최소화된 컨테이너에서는 서비스·네트워크·웹 항목(Finger·NFS·NIS·DNS·SNMP·FTP·메일·Telnet·Apache/IIS)이 **대부분 skip**으로 나온다. 이는 정상이며, skip 목록은 감사 근거로도 쓰인다.
- 베팅(해석 품질)이 드러나려면 개발은 **fail이 실제로 나오는 계정·파일권한 축(U-01~U-33)을 먼저** 다듬어야 한다 → PRD의 확장 순서(계정·파일권한 → 서비스/패치/로그 → 웹)와 일치.
- 데모 우선순위: **해석 품질 우선**(사용자 확정). 표준 전수 커버리지는 조연.
