# AI 기반 인프라 보안 스캐너 MVP PRD

**문서 버전:** 1.0  
**최종 수정:** 2026-06-24  
**팀 규모:** 3명 (개발 초심자)  
**예상 개발 기간:** 20일 (실제 활용 가능 시간: 10~12일)

## 핵심 네러티브

리눅스 서버 인프라의 보안 점검을 담당하는 보안팀은 매일 업데이트되는 CVE를 수동으로 모니터링하면서 취약점 발견이 늦어지고, 주요정보기반시설 보안 가이드에 따른 수동 점검은 시간이 오래 걸리고 오류 가능성이 높다는 문제에 직면해 있습니다. 특히 인프라가 수백~수천 대의 서버로 확장되면서 수동 관리는 현실적으로 불가능해졌습니다.

이 시스템은 **"보안 점검을 완전히 자동화할 수 있다"**는 관점에서, Ansible로 대규모 서버의 정보를 동시에 수집하고, Claude AI를 통해 지능형 취약점 분석과 조치 방안을 제시하며, 웹 대시보드로 시각화함으로써 문제를 해결합니다.

결과적으로 보안담당자는 반복적인 수작업에서 해방되고, 대규모 인프라 전체에서 빠르게 취약점을 찾아 우선순위를 정하고 경영진에게 명확한 보안 상태를 보고할 수 있게 됩니다.

### 핵심 요소

- **대상**: 리눅스 서버 인프라의 보안 점검을 담당하는 보안팀과 보안담당자
- **진짜 문제**: 수동으로 이루어지는 보안 점검으로 인한 취약점 탐지 지연, 높은 오류 가능성, 규모 확장 불가
- **맥락·타이밍**: CVE 매일 업데이트, 인프라 규모 확장, 규정 강화로 수동 관리 불가능한 상황 발생
- **핵심 관점**: 자동화(Ansible) + AI 분석(Claude) + 시각화(대시보드)의 결합으로 보안 점검 완전 자동화 가능
- **성공의 모습**: 점검 시간 80% 단축, 우선순위 명확화, 대규모 인프라 관리 가능

## 문제 정의

### 현재 상황

리눅스 서버의 보안 점검이 완전히 수동으로 이루어지고 있어 다음과 같은 문제가 발생합니다:

- **취약점 탐지 지연**: 매일 업데이트되는 CVE(Common Vulnerabilities and Exposures)를 수동으로 모니터링하고 있어 취약점 발견이 늦음
- **비효율적인 검증 프로세스**: 주요정보기반시설 보안 가이드에 따른 수동 점검이 시간이 많이 걸리고 오류 가능성 높음
- **규모 확장 불가**: 수백 대에서 수천 대의 서버를 수동으로 관리하는 것이 현실적으로 불가능
- **자동화 도구 부재**: 보안팀이 대규모 인프라 전체에서 취약점을 빠르게 찾고 보고할 수 있는 자동화 도구 없음

### 이로 인한 영향

- 심각한 보안 위협을 빠르게 대응하지 못함
- 인프라 전체에 대한 불완전한 보안 커버리지
- 보안팀의 반복적이고 비효율적인 수작업 증가
- 경영진에 대한 보안 상태 현황 보고 및 추적의 어려움

## 솔루션

리눅스 서버 인프라의 보안 점검을 자동화하는 **AI 기반 보안 스캐너 MVP**를 구축합니다. 이 시스템은 다음 3가지 핵심 기능을 제공합니다:

1. **자동화된 서버 정보 수집** - Ansible을 활용하여 여러 리눅스 서버에서 OS 정보 자동 추출
2. **AI 기반 취약점 분석** - Claude AI를 이용한 지능형 취약점 분석 및 조치 방안 제시
3. **웹 대시보드 시각화** - 취약점 현황을 한눈에 파악하고 우선순위를 정할 수 있는 대시보드

이 MVP는 다음에 집중합니다:
- 대규모 서버 자산을 자동으로 수집하고 관리
- AI 분석을 통한 정확한 취약점 분류 및 심각도 판단
- 직관적인 대시보드를 통한 취약점 추적 및 시각화

## 사용자 스토리

### 서버 자산 관리

1. 보안담당자로서, 서버 정보(IP, 호스트명, OS, 계정)가 포함된 CSV 파일을 한 번에 임포트하고 싶습니다. 그래야 수십 대의 서버를 빠르게 등록할 수 있습니다.

2. 보안담당자로서, 업로드한 파일에서 필수 정보(IP, 호스트명, OS 타입, 계정)가 모두 있는지 자동으로 검증받고 싶습니다. 그래야 유효한 서버만 등록됩니다.

3. 보안담당자로서, 임포트 중 오류가 있는 행의 정확한 원인을 알고 싶습니다. 그래야 파일을 수정하고 다시 업로드할 수 있습니다.

4. 보안담당자로서, 개별 서버를 추가하거나 수정, 삭제할 수 있기를 원합니다. 그래야 인프라 변화에 맞춰 서버 목록을 유지할 수 있습니다.

5. 보안담당자로서, 각 서버의 마지막 점검 시간과 현재 상태를 보고 싶습니다. 그래야 어느 서버를 최근에 검사했는지 알 수 있습니다.

6. 보안담당자로서, 각 서버의 상태를 시각적으로 구분(정상/경고/위험)하고 싶습니다. 그래야 문제가 있는 서버를 빠르게 찾을 수 있습니다.

7. 시스템관리자로서, 서버의 계정 정보가 안전하게 암호화되어 저장되기를 원합니다. 그래야 계정 정보가 보호됩니다.

8. 시스템관리자로서, 암호화된 계정 정보에는 관리자만 접근할 수 있기를 원합니다. 그래야 민감한 정보가 기밀로 유지됩니다.

### 서버 정보 자동 수집

9. 보안담당자로서, 시스템이 SSH를 통해 리눅스 서버의 정보를 자동으로 수집하기를 원합니다. 그래야 매번 직접 로그인할 필요가 없습니다.

10. 보안담당자로서, OS 버전, 커널 버전, CPU, 메모리, 디스크 등 하드웨어 정보를 수집하고 싶습니다. 그래야 완전한 서버 정보를 보유합니다.

11. 보안담당자로서, 서버에 설치된 패키지 목록과 버전 정보를 수집하고 싶습니다. 그래야 오래되었거나 취약한 소프트웨어를 찾을 수 있습니다.

12. 보안담당자로서, SSH 설정, 방화벽 규칙, SELinux 상태, 사용자 권한 등 보안 설정을 수집하고 싶습니다. 그래야 보안 태세를 평가할 수 있습니다.

13. 보안담당자로서, 현재 실행 중인 서비스 목록을 수집하고 싶습니다. 그래야 어떤 서비스가 노출될 위험이 있는지 알 수 있습니다.

14. 보안담당자로서, 수집한 정보에서 비밀번호, API 키, 개인 키 등 민감한 정보는 자동으로 가려지기를 원합니다. 그래야 보안 사고를 예방합니다.

15. 보안담당자로서, 여러 서버를 동시에 점검하고 싶습니다(설정 가능, 기본값: 5~10개). 그래야 전체 점검 시간이 단축됩니다.

16. 보안담당자로서, 점검에 실패한 서버의 원인을 상세히 알고 싶습니다. 그래야 연결이나 권한 문제를 해결할 수 있습니다.

17. 보안담당자로서, 실패한 점검이 자동으로 재시도되기를 원합니다(최대 3회, 30초 간격). 그래야 일시적인 네트워크 오류로 점검이 실패하지 않습니다.

18. 보안담당자로서, 각 서버의 점검이 5분 안에 완료되지 않으면 자동으로 중단되기를 원합니다. 그래야 끊긴 연결이 다른 점검을 막지 않습니다.

19. 보안담당자로서, 전체 점검 후 성공/실패 결과를 요약해서 보고 싶습니다. 그래야 점검 완료 상황을 빠르게 파악합니다.

### AI 기반 취약점 분석

20. 보안담당자로서, 수집한 서버 정보를 이용해 알려진 취약점을 자동으로 찾고 싶습니다. 그래야 어떤 취약점이 영향을 미치는지 알 수 있습니다.

21. 보안담당자로서, 찾은 취약점을 심각도별로(매우심각/높음/중간/낮음) 분류하고 싶습니다. 그래야 어떤 것부터 해결해야 할지 결정할 수 있습니다.

22. 보안담당자로서, 각 취약점이 어느 서버에 영향을 미치는지 알고 싶습니다. 그래야 영향 범위를 파악합니다.

23. 보안담당자로서, 각 취약점의 해결 방법을 한국어로 제시받고 싶습니다. 그래야 팀원들이 쉽게 이해하고 실행할 수 있습니다.

24. 보안담당자로서, 취약점 정보에 CVE 번호, 영향받는 패키지 버전, 원인 분석 등이 포함되기를 원합니다. 그래야 의사결정에 필요한 정보를 모두 가집니다.

25. 보안담당자로서, AI가 단순한 매칭이 아닌 지능형 분석을 제공하기를 원합니다. 그래야 더 정확한 인사이트를 얻습니다.

### 취약점 데이터베이스 연동

26. 보안담당자로서, 시스템이 NVD(국가 취약점 데이터베이스)의 최신 정보를 주기적으로 업데이트하기를 원합니다. 그래야 최신 취약점 정보로 점검합니다.

27. 보안담당자로서, NVD API를 사용할 수 없을 때 캐시된 이전 데이터로 동작하기를 원합니다. 그래야 외부 장애 상황에서도 계속 점검할 수 있습니다.

28. 보안담당자로서, 수집한 패키지 버전과 알려진 취약점이 자동으로 연결되기를 원합니다. 그래야 수동 작업 없이 관련 취약점을 찾습니다.

### 대시보드 및 시각화

29. 보안담당자로서, 메인 대시보드에서 전체 서버 수, 정상/경고/위험 서버 현황을 한눈에 보고 싶습니다. 그래야 인프라 보안 상태를 빠르게 파악합니다.

30. 보안담당자로서, 취약점 심각도별 분포(매우심각/높음/중간/낮음)를 보고 싶습니다. 그래야 전체 위험 상황을 이해합니다.

31. 보안담당자로서, 취약점 분포를 파이 차트로 시각화해서 보고 싶습니다. 그래야 심각한 취약점의 비중을 직관적으로 이해합니다.

32. 보안담당자로서, 모든 서버의 목록(이름, IP, OS 버전, 마지막 점검 시간)을 볼 수 있기를 원합니다. 그래야 개별 서버의 상세 정보를 확인할 수 있습니다.

33. 보안담당자로서, 서버 목록을 상태별로 필터링하고 싶습니다. 그래야 문제가 있는 서버에 집중할 수 있습니다.

34. 보안담당자로서, 서버를 클릭하면 그 서버의 모든 취약점을 볼 수 있기를 원합니다. 그래야 개별 서버의 위험을 파악합니다.

35. 보안담당자로서, 각 취약점의 상세 정보(CVE 번호, 제목, 심각도, CVSS 점수, 영향받는 서버, 해결 방법)를 보고 싶습니다. 그래야 대응을 계획할 수 있습니다.

36. 보안담당자로서, 취약점을 심각도로 필터링하고 싶습니다. 그래야 가장 심각한 것부터 처리할 수 있습니다.

37. 보안담당자로서, 취약점 상세 정보에서 NVD 공식 페이지로 이동할 수 있기를 원합니다. 그래야 더 자세한 정보를 얻을 수 있습니다.

38. 보안담당자로서, 대시보드가 자동으로 새로고침되기를 원합니다(1분 간격). 그래야 최신 정보를 계속 볼 수 있습니다.

39. 보안담당자로서, 필요하면 수동으로 새로고침 버튼을 눌러서 즉시 업데이트받고 싶습니다. 그래야 자동 새로고침을 기다릴 필요가 없습니다.

40. 시스템관리자로서, 모든 서버에 접근하고 시스템 설정을 변경할 수 있길 원합니다. 그래야 완전한 관리 권한을 갖습니다.

41. 뷰어로서, 내가 담당하는 팀/부서의 서버만 보고 싶습니다. 그래야 관련된 것만 집중해서 봅니다.

## 기술 결정사항

### 사용 기술

- **백엔드**: Python (FastAPI) - REST API 및 시스템 오케스트레이션
- **프론트엔드**: React - 웹 기반 대시보드
- **데이터베이스**: PostgreSQL - 데이터 저장소
- **서버 정보 수집**: Ansible - SSH 기반 원격 명령 실행
- **AI 분석**: Claude API - 취약점 분석 및 조치 방안 생성
- **취약점 정보**: NVD API - 공식 취약점 데이터 수집
- **보안**: AES-256 암호화(계정 정보), bcrypt(사용자 비밀번호)
- **배포**: Docker 또는 로컬 서버

### 시스템 구조

3계층 아키텍처로 설계합니다:

1. **프레젠테이션 계층(React)**: 취약점 조회 및 서버 관리 웹 대시보드
2. **API 계층(FastAPI)**: 인증, 데이터 관리, Ansible 제어를 담당하는 REST API
3. **데이터 계층(PostgreSQL)**: 서버, 계정, 점검 결과, 취약점 정보를 저장

별도의 **Ansible 모듈**은 SSH를 통해 리눅스 서버와 통신합니다. **CVE 모듈**은 NVD API에서 취약점 정보를 가져오고 캐싱합니다.

### 데이터베이스 설계

**servers**: 서버 메타데이터(IP, 호스트명, OS, 설명), 마지막 점검 시간, 상태 추적

**credentials**: 암호화된 계정명, 비밀번호, SSH 키. 환경변수 `INFRA_SECURITY_MASTER_KEY`에서 암호화 키를 로드하며, 관리자만 접근 가능

**scan_results**: 점검 실행 기록, 수집한 OS 정보(패키지, 보안 설정), 민감 정보는 마스킹하여 저장

**vulnerabilities**: 각 점검에서 발견된 CVE 정보, CVSS 점수, 심각도, AI 생성 조치 방법

**vulnerability_mappings**: 패키지 버전과 취약점을 연결하여 영향받는 서버 추적

### 점검 프로세스

1. 사용자가 대시보드에서 점검 시작
2. 백엔드가 Ansible을 통해 각 서버에 SSH로 연결 (동시 처리: 5~10개)
3. 수집 스크립트가 다음 정보를 추출:
   - 시스템 정보(OS, 커널, CPU, 메모리, 디스크)
   - 설치된 패키지 목록(버전 포함)
   - 보안 설정(방화벽, SSH, SELinux, 권한)
   - 실행 중인 서비스
4. 민감한 정보는 자동으로 가림
5. 결과를 데이터베이스에 저장
6. 패키지 정보를 AI 분석 모듈로 전송

### AI 분석 과정

1. 점검 결과에서 패키지 목록 추출
2. Claude API를 호출하며 다음 정보 제공:
   - 수집한 서버 정보
   - NVD 캐시 데이터
   - 보안 설정 상세 정보
3. Claude가 반환:
   - 영향받는 CVE
   - 심각도(CVSS 점수 기반)
   - 한국어 영향도 분석
   - 한국어 해결 방법
4. AI 응답을 파싱하여 취약점 레코드 생성
5. 영향받는 서버와 연결
6. 서버의 최고 심각도로 상태 업데이트

### 계정 정보 보안

- 환경변수 `INFRA_SECURITY_MASTER_KEY`에 암호화 키 저장
- 모든 계정 정보는 AES-256으로 암호화하여 저장
- 점검 중에만 메모리에서 복호화
- 관리자는 암호화된 형태만 볼 수 있음
- 로그나 오류 메시지에 계정 정보 노출 안 함

### 안정성 기능

- 동시 처리: 5~10개 서버 (설정 가능)
- 서버당 타임아웃: 5분
- 자동 재시도: 3회, 30초 간격
- 실패한 서버는 상세 오류 기록, 나머지는 계속 진행
- 오프라인 모드: NVD API 불가 시 캐시된 데이터 사용

### API 명세

주요 엔드포인트:
- `POST /api/servers/import` — CSV/Excel/JSON에서 서버 임포트
- `GET /api/servers` — 서버 목록
- `POST /api/servers/{id}/scan` — 서버 점검 시작
- `GET /api/scans/{id}/results` — 점검 결과 조회
- `GET /api/vulnerabilities` — 취약점 목록(필터링 가능)
- `GET /api/vulnerabilities/{cve_id}` — 취약점 상세정보
- `GET /api/dashboard/metrics` — 대시보드 수치
- `POST /api/auth/login` — 로그인

### 권한 관리

- **관리자**: 모든 서버 접근, 점검 실행, 설정 변경, 계정 정보 접근
- **뷰어**: 담당 팀/부서의 서버와 취약점만 읽기 전용 접근
- 인증: ID/비밀번호, JWT 토큰으로 세션 관리

### 테스트 전략

포괄적인 테스트를 위해 외부 종속성(Ansible, Claude API, NVD API)을 모킹합니다:

1. **Ansible 모킹**: 실제 서버 없이 수집 로직 검증
2. **Claude API 모킹**: API 비용 없이 분석 파이프라인 검증
3. **NVD API 모킹**: 캐시된 테스트 데이터로 통합 검증
4. **데이터베이스 통합**: 실제 PostgreSQL로 전체 흐름 검증

## 테스트 계획

### 테스트 원칙

테스트는 실제 시스템 동작(API 응답, 데이터베이스 상태, 화면 상호작용)을 검증합니다. 외부 서비스는 모킹하여 안정적이고 빠른 테스트를 만듭니다.

### 테스트 범위

1. **서버 자산 관리**
   - 파일 임포트 및 검증(CSV, Excel, JSON)
   - 필수 정보 검증
   - 오류 처리 및 부분 성공
   - CRUD 작업

2. **정보 수집**
   - Ansible 로직 검증
   - 동시 처리 확인
   - 재시도 동작 검증
   - 타임아웃 처리
   - 오류 로깅
   - 민감 정보 마스킹

3. **AI 분석**
   - Claude API 연동
   - 심각도 분류
   - 한국어 출력 검증
   - 오류 처리

4. **취약점 데이터**
   - NVD API 통합
   - 캐시 관리
   - 패키지 매칭

5. **API 계층**
   - 인증/권한 부여
   - 요청 검증
   - 응답 형식
   - 오류 처리

6. **대시보드**
   - 메트릭 데이터 검색
   - 필터링/정렬
   - 자동 새로고침

### 테스트 시나리오

**서버 관리**:
- 유효한 CSV 임포트
- 필드 누락된 CSV 처리
- 유효/무효 혼합 처리
- 계정 정보 암호화/복호화
- 서버 추가/수정/삭제

**정보 수집**:
- 5~10개 서버 동시 수집
- 네트워크 오류 및 재시도
- 실패한 서버 처리
- 민감 정보 마스킹 확인
- 병렬 처리 검증

**취약점 분석**:
- 패키지와 CVE 매칭
- 심각도 분류 정확성
- 한국어 조치 방법 생성
- 불완전한 데이터 처리

**통합 테스트**:
- 전체 흐름(임포트 → 수집 → 분석 → 표시)
- 여러 서버 성공/실패 혼합
- 대시보드 데이터 정확성
- API 데이터 일관성

## 참고사항

### 개발 방식

- **Claude Code 활용**: Ansible 및 AI 통합 로직 빠른 구현
- **빠른 프로토타이핑**: 핵심 기능 중심, 최소한의 완성도, 거친 UI 허용
- **병렬 개발**: 3명 팀의 역할 분담
  - 1명: 백엔드 + Ansible
  - 1명: AI 분석 + CVE 통합
  - 1명: React 프론트엔드 + 대시보드

### 기술 조건

- 테스트 환경: 5~10개 리눅스 VM 또는 서버
- 취약점 정보: NVD 공개 API(무료)
- AI: Claude API(예상 비용 $5~10)
- 네트워크: SSH 포트 22 개방 필요
- 팀 수준: 개발 초심자(Ansible 학습: 약 1시간)

### 주요 위험 요소 및 대응

1. **Ansible 학습 곡선**: Claude Code의 자동 스크립트 생성으로 단축
2. **API 비용**: 테스트는 모킹 데이터 사용으로 최소화
3. **테스트 서버**: 필요 시 클라우드 VM으로 신속 프로비저닝
4. **일정 압박**: 엄격한 범위 관리와 병렬 개발로 극복

### 기본 가정

- 리눅스 서버는 비밀번호 또는 SSH 키로 접근 가능
- NVD API는 개발 기간 동안 정상 작동
- PostgreSQL 서버 사용 가능
- 팀이 10~12일 동안 집중 개발 가능
