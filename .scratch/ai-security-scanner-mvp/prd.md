# AI 기반 인프라(OS) 보안 점검 도구 - MVP PRD

**문서 버전:** 1.0  
**최종 수정:** 2026-06-24  
**팀 규모:** 3명 (개발 초심자)  
**기간:** 20일 (실제 가용 시간: 10~12일)

---

## 문제 정의

**현재 상황:**

인프라(OS) 보안 점검이 현재 수동으로 진행되고 있어 운영 부담이 매우 큽니다:

- **수동 모니터링:** 매일 업데이트되는 CVE를 수동으로 모니터링해야 하므로 취약점 탐지가 지연됨
- **비효율적 점검:** 주요정보기반시설 취약점 가이드에 따른 보안 점검이 시간이 오래 걸리고 오류 가능성 높음
- **확장성 문제:** IDC 환경의 수백~수천 대 서버를 수동 프로세스로 관리하기 불가능
- **응답 지연:** 보안팀이 대규모 서버 플릿 전체에서 취약점을 빠르게 식별하고 보고할 자동화 도구 부족

**영향:**

- 심각한 취약점 탐지 지연
- 인프라 전체에 대한 불완전한 커버리지
- 보안팀의 높은 수동 작업 부하
- 경영진에 대한 취약점 상태 보고 및 추적의 어려움

---

## 솔루션

Linux 서버 플릿 전체의 OS 보안 점검을 자동화하는 **AI 기반 보안 스캐너 MVP**를 구축합니다. 시스템은 다음을 수행합니다:

1. **Ansible을 통한 자동 OS 정보 수집** - 여러 Linux 서버에서 OS 정보 자동 추출
2. **Claude AI를 이용한 취약점 분석** - 설치된 패키지와 보안 설정값 기반 취약점 분석
3. **웹 대시보드로 결과 제시** - 취약점 추적 및 조치 우선순위 결정 용이

이 MVP는 AI 기반 보안 자동화의 가능성을 입증하기 위해 3가지 핵심 기능에 집중합니다:
- 자동화된 OS 자산 관리 및 정보 수집
- AI 기반 취약점 분석 및 심각도 분류
- 취약점 추적 및 시각화 웹 대시보드

## 사용자 스토리

### 자산 관리 및 인벤토리

1. 보안담당자로서, 서버 상세정보(IP, 호스트명, OS, 자격증명)가 포함된 CSV 파일을 임포트하고 싶습니다. 그래야 수동 입력 없이 여러 서버를 빠르게 등록할 수 있습니다.

2. 보안담당자로서, 임포트 파일의 필수 필드(IP, 호스트명, OS 타입, 계정정보)를 시스템이 검증하기를 원합니다. 그래야 유효한 서버 레코드만 추가됩니다.

3. 보안담당자로서, 임포트 중 잘못된 행에 대한 상세 오류 메시지를 보고 싶습니다. 그래야 데이터를 수정하고 다시 임포트할 수 있습니다.

4. 보안담당자로서, 자산 인벤토리에서 개별 서버를 추가/수정/삭제할 수 있길 원합니다. 그래야 인프라 변화에 따라 서버 목록을 최신 상태로 유지할 수 있습니다.

5. 보안담당자로서, 각 서버의 마지막 스캔 날짜와 결과 상태를 보고 싶습니다. 그래야 어떤 서버가 최근에 평가되었는지 알 수 있습니다.

6. 보안담당자로서, 각 서버의 시각적 상태 표시(정상/경고/위험)를 보고 싶습니다. 그래야 문제 영역을 빠르게 식별할 수 있습니다.

7. 시스템 관리자로서, 시스템이 서버 자격증명을 안전하게 저장하기를 원합니다. 그래야 인증 정보가 무단 접근으로부터 보호됩니다.

8. 시스템 관리자로서, 관리자 사용자만 암호화된 자격증명에 접근하기를 원합니다. 그래야 민감한 인증 데이터가 기밀로 유지됩니다.

### OS 정보 추출

9. 보안담당자로서, 시스템이 SSH를 통해 Linux 서버에서 OS 정보를 자동으로 수집하기를 원합니다. 그래야 각 서버에 수동으로 로그인할 필요가 없습니다.

10. 보안담당자로서, 시스템이 OS 버전, 커널 버전, CPU, 메모리, 디스크 정보를 추출하기를 원합니다. 그래야 완전한 하드웨어 및 소프트웨어 인벤토리를 갖출 수 있습니다.

11. 보안담당자로서, 시스템이 설치된 패키지와 버전 목록을 검색하기를 원합니다. 그래야 오래되었거나 취약한 소프트웨어를 식별할 수 있습니다.

12. 보안담당자로서, 시스템이 SSH 설정값, 방화벽 규칙, SELinux 상태, 사용자 권한을 수집하기를 원합니다. 그래야 보안 설정 태세를 평가할 수 있습니다.

13. 보안담당자로서, 시스템이 활성 서비스 상태 정보를 수집하기를 원합니다. 그래야 어떤 서비스가 실행 중이고 노출될 수 있는지 알 수 있습니다.

14. 보안담당자로서, 시스템이 수집된 데이터에서 민감한 정보(비밀번호, API 키, 개인 키, 토큰)를 마스킹하기를 원합니다. 그래야 자격증명이 노출되지 않습니다.

15. 보안담당자로서, 시스템이 여러 서버를 병렬로 스캔하기를 원합니다(설정 가능, 기본값: 5~10개). 그래야 평가가 빠르게 완료됩니다.

16. 보안담당자로서, 실패한 스캔이 상세 오류 메시지와 함께 기록되기를 원합니다. 그래야 연결 또는 접근 문제를 해결할 수 있습니다.

17. 보안담당자로서, 시스템이 실패한 스캔을 30초 간격으로 최대 3회 자동으로 재시도하기를 원합니다. 그래야 임시 네트워크 문제로 스캔이 실패하지 않습니다.

18. 보안담당자로서, 시스템이 개별 스캔을 5분 후에 타임아웃하기를 원합니다. 그래야 중단된 연결이 다른 평가를 차단하지 않습니다.

19. 보안담당자로서, 일괄 스캔 후 성공한 서버와 실패한 서버의 개수를 보여주는 요약 보고서를 보고 싶습니다. 그래야 스캔 완료도를 확인할 수 있습니다.

### AI 기반 취약점 분석

20. 보안담당자로서, 시스템이 수집된 OS 데이터를 CVE 데이터베이스와 비교 분석하기를 원합니다. 그래야 서버에 영향을 미치는 알려진 취약점을 식별할 수 있습니다.

21. 보안담당자로서, 취약점이 CVSS v3.1 스코어를 기반으로 심각도(Critical, High, Medium, Low)로 분류되기를 원합니다. 그래야 조치 노력에 우선순위를 정할 수 있습니다.

22. 보안담당자로서, 시스템이 각 CVE의 영향을 받는 서버를 식별하기를 원합니다. 그래야 각 취약점의 영향 범위를 알 수 있습니다.

23. 보안담당자로서, 각 취약점이 한국어로 조치 권장사항을 포함하기를 원합니다. 그래야 한국어를 사용하는 팀이 조치를 이해하고 구현할 수 있습니다.

24. 보안담당자로서, 각 취약점이 CVE ID, 영향받는 패키지 버전, 근본 원인 분석을 포함하기를 원합니다. 그래야 의사결정을 위한 완전한 정보를 가질 수 있습니다.

25. 보안담당자로서, 취약점 분석이 Claude AI를 사용하여 단순 CVE 매칭 이상의 지능형 인사이트를 제공하기를 원합니다. 그래야 AI의 문맥적 이해의 이점을 누릴 수 있습니다.

### CVE 데이터베이스 연동

26. 보안담당자로서, 시스템이 NVD(국가 취약점 데이터베이스)와 정기적으로 동기화하기를 원합니다. 그래야 최신 알려진 취약점을 기준으로 평가할 수 있습니다.

27. 보안담당자로서, NVD API를 사용할 수 없을 때 시스템이 캐시된 CVE 데이터로 돌아가기를 원합니다. 그래야 외부 서비스 장애 중에도 스캔을 계속할 수 있습니다.

28. 보안담당자로서, CVE 데이터가 수집된 패키지 버전과 자동으로 매칭되기를 원합니다. 그래야 관련 취약점이 수동 작업 없이 식별됩니다.

### 웹 대시보드 및 시각화

29. 보안담당자로서, KPI 메트릭(전체 서버 수, 정상/경고/위험 개수)을 포함한 메인 대시보드를 보고 싶습니다. 그래야 인프라 상태를 높은 수준에서 한눈에 볼 수 있습니다.

30. 보안담당자로서, 심각도별 분포 메트릭(Critical/High/Medium/Low 취약점 개수)을 보고 싶습니다. 그래야 전체 취약점 상황을 이해할 수 있습니다.

31. 보안담당자로서, 심각도별 취약점 분포를 보여주는 파이 차트를 보고 싶습니다. 그래야 심각한 것과 심각하지 않은 이슈의 비율을 시각화할 수 있습니다.

32. 보안담당자로서, 모든 서버의 목록(이름, IP, OS 버전, 마지막 스캔 시간)을 보고 싶습니다. 그래야 개별 서버 상태를 상세히 확인할 수 있습니다.

33. 보안담당자로서, 서버 목록을 상태(정상/경고/위험)로 필터링할 수 있기를 원합니다. 그래야 문제가 있는 서버에 집중할 수 있습니다.

34. 보안담당자로서, 서버를 클릭하여 그 서버에 영향을 미치는 모든 취약점을 보고 싶습니다. 그래야 전체 위험 프로필을 이해할 수 있습니다.

35. 보안담당자로서, 상세한 취약점 정보(CVE ID, 제목, 심각도, CVSS 스코어, 영향받는 서버, 조치 단계)를 보고 싶습니다. 그래야 수정을 계획하는 데 필요한 모든 정보를 가질 수 있습니다.

36. 보안담당자로서, 취약점을 심각도 수준으로 필터링할 수 있기를 원합니다. 그래야 Critical과 High 취약점에 우선순위를 정할 수 있습니다.

37. 보안담당자로서, CVE 링크를 클릭하여 NVD의 공식 취약점 상세정보를 볼 수 있기를 원합니다. 그래야 권위있는 정보에 접근할 수 있습니다.

38. 보안담당자로서, 대시보드가 매분마다 자동으로 새로고침되기를 원합니다. 그래야 거의 실시간의 취약점 상태를 볼 수 있습니다.

39. 보안담당자로서, 필요할 때 대시보드를 수동으로 새로고침할 수 있기를 원합니다. 그래야 자동 새로고침을 기다리지 않고 즉시 업데이트를 받을 수 있습니다.

40. 관리자로서, 모든 서버에 접근하고 시스템 설정을 수정할 수 있길 원합니다. 그래야 완전한 관리 권한을 갖습니다.

41. 뷰어로서, 내 팀/부서에 할당된 서버만 보길 원합니다. 그래야 관련 인프라에 집중할 수 있습니다.

## 구현 결정사항

### 기술 스택

- **백엔드:** Python (FastAPI) - REST API 및 오케스트레이션
- **프론트엔드:** React - 웹 대시보드
- **데이터베이스:** PostgreSQL - 데이터 지속성
- **OS 정보 수집:** Ansible - SSH 기반 원격 실행 (Linux 서버)
- **AI 분석:** Claude API - 취약점 분석 및 권장사항 생성
- **CVE 데이터:** NVD API - 실시간 CVE 정보
- **자격증명 암호화:** AES-256 (민감 정보), bcrypt (사용자 비밀번호)
- **배포:** Docker 컨테이너 또는 로컬 서버

### 핵심 아키텍처

시스템은 3계층 아키텍처를 따릅니다:

1. **프레젠테이션 계층(React):** 취약점 조회 및 서버 관리 웹 대시보드
2. **API 계층(FastAPI):** 인증, CRUD 작업, Ansible 오케스트레이션을 담당하는 REST API
3. **데이터 계층(PostgreSQL):** 서버, 자격증명, 스캔 결과, 취약점 정보를 저장하는 중앙 데이터베이스

별도의 **Ansible 실행기** 모듈은 SSH를 통해 Linux 서버와 인터페이스하여 OS 정보를 수집합니다. **CVE 커넥터**는 NVD API와 인터페이스하여 취약점 데이터를 가져오고 캐싱합니다.

### 데이터베이스 스키마 결정사항

**서버 테이블:**
- 서버 메타데이터 저장 (IP, 호스트명, OS 타입, 설명)
- 마지막 스캔 타임스탬프 및 전체 상태 추적
- 암호화된 자격증명 참조

**자격증명 테이블:**
- 암호화된 사용자명, 비밀번호 또는 SSH 키 저장
- 환경변수 `INFRA_SECURITY_MASTER_KEY`에서 마스터 암호화 키 로드
- 관리자 사용자만 접근 가능; 활성 스캔 중에만 복호화

**스캔 결과 테이블:**
- 각 OS 정보 수집의 타임스탐프 레코드
- 수집된 데이터 저장 (OS 버전, 패키지, 보안 설정)
- 민감한 필드(비밀번호, 토큰) 저장 전 마스킹

**취약점 테이블:**
- 스캔 결과와 연결된 CVE 정보
- CVSS 스코어 및 심각도 분류
- AI 생성 조치 권장사항

**취약점 매핑 테이블:**
- 패키지 버전 기반 취약점과 영향받는 서버 연결
- 조치 우선순위 및 상태 추적

### OS 정보 수집 흐름

1. 사용자가 웹 대시보드에서 스캔 시작
2. 백엔드가 스캔 작업 생성하고 Ansible 실행기 트리거
3. Ansible이 각 서버에 SSH로 연결 (설정 가능한 병렬처리: 기본값 5~10개 동시)
4. 원격 수집 스크립트가 다음 정보 추출:
   - 시스템 정보 (OS 버전, 커널, CPU, 메모리, 디스크)
   - 패키지 인벤토리 (버전 포함)
   - 보안 설정 (방화벽, SSH 설정, SELinux, 권한)
   - 실행 중인 서비스
5. 수집 중 민감한 정보 마스킹
6. 결과를 타임스탐프와 함께 데이터베이스에 저장
7. 마스킹된 패키지 목록을 AI 분석기로 전송

### AI 분석 파이프라인

1. 각 스캔 결과에서 패키지 목록 및 보안 설정 추출
2. 다음을 포함한 구조화된 프롬프트로 Claude API 호출:
   - 수집된 OS 및 패키지 정보
   - 알려진 CVE 데이터베이스 (NVD 캐시)
   - 시스템 설정 상세정보
3. Claude가 다음을 반환하도록 분석:
   - CVE ID와 함께 식별된 취약점
   - 심각도 분류 (CVSS 스코어 기반)
   - 한국어 영향도 분석
   - 한국어 조치 권장사항
4. Claude 응답 파싱 및 취약점 레코드 생성
5. 영향받는 서버에 취약점 연결
6. 최대 심각도를 기반으로 서버 상태 업데이트

### 자격증명 보안 모델

- 마스터 암호화 키를 환경변수로 저장: `INFRA_SECURITY_MASTER_KEY`
- AES-256을 사용한 모든 자격증명 암호화 저장
- 활성 스캔 중에만 메모리에서 복호화
- 관리자 사용자는 암호화된 표현만 볼 수 있음
- 복호화된 자격증명은 로그되거나 오류 메시지에 노출되지 않음
- 사용 직후 메모리에서 자격증명 제거

### 스캔 복원력

- 설정 가능한 동시성으로 병렬 실행 (기본값: 5~10개 서버)
- 서버당 5분 타임아웃
- 자동 재시도 로직: 3회 시도, 30초 간격
- 실패한 서버는 상세 오류 메시지와 함께 기록; 나머지 서버는 계속 스캔
- API 다운타임 복원력: 오프라인 매칭을 위해 마지막 CVE 데이터베이스 캐싱

### API 계약

**핵심 REST 엔드포인트:**

- `POST /api/servers/import` — CSV/Excel/JSON에서 서버 임포트
- `GET /api/servers` — 상태와 함께 모든 서버 목록
- `POST /api/servers/{id}/scan` — 서버 스캔 트리거
- `GET /api/scans/{id}/results` — 스캔 결과 조회
- `GET /api/vulnerabilities` — 필터링 가능한 취약점 목록
- `GET /api/vulnerabilities/{cve_id}` — CVE 상세정보
- `GET /api/dashboard/metrics` — 대시보드 KPI 메트릭
- `POST /api/auth/login` — 사용자 인증

### 접근 제어

- **관리자 역할:** 모든 서버에 완전 접근, 스캔 트리거, 설정 수정, 자격증명 접근
- **뷰어 역할:** 팀/부서에 할당된 서버/취약점에 대한 읽기 전용 접근
- ID/비밀번호로 인증; 세션 관리를 위한 JWT 토큰

### 테스트 지점

**통합 테스트를 위한 고수준 지점:**

1. **Ansible 통합 지점:** 실제 Linux 서버 없이 OS 수집 흐름을 테스트하기 위한 Ansible 실행기 모킹
   - 스캔 로직, 병렬 실행, 재시도 로직, 오류 처리 테스트 가능
   - 현실적인 OS 정보를 나타내는 픽스처 데이터 사용

2. **Claude API 통합 지점:** API 할당량을 소비하지 않고 취약점 분석 테스트하기 위한 Claude API 모킹
   - 분석 파이프라인 테스트 가능
   - 알려진 시나리오에 대한 사전 정의된 응답 픽스처 사용

3. **NVD API 통합 지점:** CVE 데이터와 데이터베이스 통합 테스트하기 위한 NVD API 모킹
   - 캐시된 픽스처 데이터 사용

4. **종단 간 데이터베이스 지점:** 실제 PostgreSQL 테스트 데이터베이스
   - 완전한 데이터 지속성 흐름 테스트
   - 서버 및 자격증명 데이터를 위한 테스트 픽스처 사용

이러한 지점들은 실제 Linux 서버나 높은 API 사용 비용 없이 포괄적인 테스트를 가능하게 합니다.

## 테스트 결정사항

### 테스트 철학

테스트는 구현 세부사항이 아닌 외부 시스템 동작(API 응답, 데이터베이스 상태, UI 상호작용)을 검증해야 합니다. 외부 종속성(Ansible, Claude API, NVD API)을 모킹하여 시스템 컴포넌트를 격리하고 테스트 시나리오를 제어합니다.

### 테스트할 모듈

1. **자산 관리 모듈**
   - 파일 임포트 및 검증 (CSV, Excel, JSON 파싱)
   - 필수 필드 검증
   - 오류 보고 및 부분 성공 처리
   - 서버 CRUD 작업

2. **OS 정보 수집 모듈**
   - Ansible 오케스트레이션 로직
   - 병렬 실행 및 동시성 제어
   - 재시도 메커니즘 동작
   - 타임아웃 처리
   - 오류 로깅 및 복구
   - 민감 데이터 마스킹

3. **AI 분석 모듈**
   - Claude API 요청/응답 처리
   - 취약점 분류 로직
   - 한국어 출력 생성
   - API 실패 시 오류 처리

4. **CVE 데이터베이스 모듈**
   - NVD API 통합
   - 캐시 무효화 및 새로고침
   - API를 사용할 수 없을 때 캐시된 데이터로 돌아가기
   - 수집된 패키지와 CVE 매칭

5. **API 계층**
   - 인증 및 권한 부여
   - 요청 검증
   - 응답 형식 지정
   - 오류 처리

6. **대시보드 통합**
   - KPI 메트릭 데이터 검색
   - 취약점 필터링 및 정렬
   - 자동 새로고침 동작

### 테스트 시나리오

**자산 관리:**
- 모든 필수 필드가 있는 유효한 CSV 임포트
- 필수 필드가 누락된 잘못된 CSV
- 유효한/잘못된 행이 혼합된 경우 (부분 성공)
- 자격증명 암호화/복호화
- 서버 생명주기 (추가/업데이트/삭제)

**OS 수집:**
- 5~10개 서버에서 성공적인 수집
- 네트워크 타임아웃 및 재시도 동작
- 실패한 서버 처리 (다른 서버는 계속)
- 민감 데이터 마스킹 검증
- 동시성 제한 내의 병렬 실행

**취약점 분석:**
- 설치된 패키지와 CVE 매칭
- 심각도 분류 정확성
- 한국어 조치 권장사항 생성
- 불완전한 패키지 데이터 처리

**통합 테스트:**
- 전체 흐름: 임포트 → 수집 → 분석 → 표시
- 성공/실패 혼합된 다중 서버 스캔
- 스캔 후 대시보드 데이터 정확성
- API 데이터 일관성

## 성공 기준

이 MVP는 다음 3가지 기능이 입증될 때 성공입니다:

### 성공 기준 1: 자산 관리 + 자동 OS 수집

- CSV 임포트로 5~10개 Linux 서버 등록 (IP, 호스트명, 계정 정보 포함)
- Ansible 기반 수집이 등록된 서버에서 OS 정보 검색
- 성공률: 80% 이상 서버 정상 스캔
- 수집된 데이터 포함: OS 버전, 패키지 (버전 포함), 보안 설정
- 실패한 서버는 상세 오류 메시지와 함께 기록

### 성공 기준 2: AI 기반 취약점 분석

- Claude API가 수집된 서버 데이터 분석
- 취약점이 심각도 수준(Critical/High/Medium/Low)으로 분류
- 한국어 조치 권장사항 제공
- CVE 데이터가 수집된 패키지와 매칭
- 영향받는 서버가 각 취약점에 대해 식별

### 성공 기준 3: 웹 대시보드 취약점 시각화

- 대시보드에서 모든 발견된 취약점 조회 가능
- 심각도별 취약점 필터링 가능
- 서버 상태가 취약점 타입 분포 표시
- 각 취약점이 다음을 표시: CVE ID, 영향받는 서버, 조치 단계
- 대시보드가 최신 데이터를 표시하도록 자동 새로고침

- **Claude Code 활용:** Ansible 스크립트 생성 및 AI 통합 로직 가속화
- **빠른 프로토타이핑:** MVP는 핵심 기능에 집중; 최소한의 폴리싱; 거친 UI는 허용
- **팀 협업:** 3명 팀이 다음과 같이 병렬 개발:
  - 백엔드 + Ansible 개발 (1명)
  - AI 분석 + CVE 통합 (1명)
  - React 프론트엔드 + 대시보드 (1명)

### 기술 제약

- 테스트 환경: 5~10개 Linux VM 또는 물리 서버
- CVE 데이터: 공개 NVD API (무료 티어)
- AI 분석: Claude API (MVP 개발 비용 예상: ~$5~10)
- 네트워크: SSH 포트 22 연결 필수
- 팀 역량: 개발 초심자; Ansible 학습 곡선 1시간 예상

### 알려진 위험

1. **Ansible 학습 곡선:** 팀이 Ansible에 익숙하지 않음; 완화: Claude Code로 스크립트 자동 생성
2. **API 비용:** 개발 중 Claude API 사용; 완화: 테스트에서 모킹 응답 사용
3. **테스트 서버 가용성:** VM 프로비저닝 필요; 인프라 가용성에 따라 결정됨
4. **타이트한 일정:** 실제 가용 10~12일; 완화: 엄격한 범위 준수, 병렬 개발

### 가정

- Linux 서버는 비밀번호 또는 키 기반 SSH 접근 가능
- NVD API가 개발 기간 동안 사용 가능
- PostgreSQL이 데이터 지속성을 위해 사용 가능
- 팀이 10~12일 동안 적극적으로 개발에 집중 가능
