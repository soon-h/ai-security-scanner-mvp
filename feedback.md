## ⚠️ 부족한 점 (위험 요인)

### 1️⃣ 개발 초심자 3명 + 회사 업무 병행
**리스크:** 실제 가용 시간 부족
- 초심자들이 학습하며 구현해야 함
- 회사 업무와의 분리가 어려울 수 있음
- **추정 실제 가용 시간:** 20일의 50~60% 정도?
- → 10~12일 분량의 업무만 가능할 가능성

**영향:** 
- Ansible 학습 + 구현에 3일 예정 → 실제로는 4~5일 소요 가능
- React 대시보드 개발 → 초보자에겐 예상보다 2~3배 오래 걸릴 가능성

### 2️⃣ AI API 활용 방안이 아직 불명확 ⛔️ **가장 큰 불확실성**
**문제:**
- SPEC에서 "AI가 리뷰 자동 처리"는 구체적으로 무엇인가?
- 취약점 분석 결과를 "AI가 자동 생성"한다는 뜻인가?
- 아니면 "자동으로 사람이 리뷰해야 할 항목을 필터링"한다는 뜻인가?

**비용 고려사항:**
- Claude API는 저렴하지만, 대량 스캔 시 누적될 수 있음
- 서버 5대 × 여러 번 스캔 = API 호출 누적
- 비용 예산이 정해지지 않으면 범위 조정 필요

**필요한 결정:**
- AI를 정확히 어느 단계에서 어떻게 쓸 것인가?
- 프롬프트는 어떻게 설계할 것인가?
- API 호출 횟수/비용 예상은?

### 3️⃣ SPEC 범위가 크다
**현재 SPEC의 기능:**
- 자산 관리 (CSV/Excel 파싱)
- OS 정보 추출 (Linux SSH + Windows WinRM)
- AI 기반 취약점 분석
- 웹 대시보드 (KPI, 차트, 취약점 조회)
- CVE 데이터베이스 연동
- 이메일 알림
- 정기 스캔 스케줄링
- 리포트 생성 (PDF/Excel)
- 조치 추적 및 상태 관리
- 역할 기반 접근 제어 (RBAC: Admin/Operator/Viewer)
- 데이터 아카이빙 (1년 이상 데이터)

**문제:**
- 초심자 3명이 회사 업무 병행하면서 **모든 기능을 20일에 구현하기는 무리**
- 특히 **React 대시보드**는 초보자에겐 생각보다 오래 걸림 (차트, 실시간 갱신 등)
- **RBAC 권한 관리**도 복잡함

### 4️⃣ Ansible 미경험
**리스크:**
- OS 정보 추출의 핵심 기술이 Ansible인데 처음 봄
- 3~4일 예정인 일정이 5~6일로 늘어날 가능성
- 초심자가 Ansible 문법 + SSH 원격 접속 + 병렬 처리를 동시에 학습해야 함

   - 프롬프트 예시 작성
   - 예상 비용 계산
   - → 이것이 가장 큰 불확실성

   - 회사 업무와의 경계를 명확히 하기
