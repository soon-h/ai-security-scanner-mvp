# AI 기반 인프라(OS) 보안 점검 도구 - MVP Spec (개선 버전)

**작성일:** 2026-06-24
**버전:** 2.0 (범위 조정 + 실제 가용 시간 기반)
**팀:** 3명 (개발 초심자)
**기간:** 20일 (실제 가용 시간 기준: 10~12일)
**목표:** "척추 한 줄" 3가지 기능으로 AI 기반 보안 점검 자동화 검증

---

## 1. 문제 정의 (원본 유지)

**현재 상황:**

- 인프라(OS) 보안 점검을 수동으로 진행 중
- 매일 업데이트되는 CVE를 수동으로 모니터링
- 주요정보기반시설 취약점 가이드를 기준으로 한 점검이 비효율적

**해결하려는 것:**

- **자동화:** OS 정보를 자동으로 추출하고 AI가 분석
- **조기 발견:** 취약점을 빠르게 식별하고 보고
- **대규모 관리:** IDC 수준의 대량 서버를 효율적으로 점검 (MVP는 5~10대 먼저)

---

## 2. 핵심 사용자 (원본 유지)

**주요 사용자:** 보안팀 (Security Engineer, Sysadmin)

- 대규모 IDC 환경의 수백~수천 대 서버를 관리
- 보안 취약점을 빠르게 파악하고 대응해야 함
- 점검 결과와 이력을 추적하고 경영진에게 보고

---

## 3. 핵심 기능 (MVP - 범위 조정)

### 3.0 자산 관리 및 인벤토리 ✅ (포함)

#### 3.0.1 자산관리대장 (Asset Inventory) 파일 지원

- **파일 형식:** CSV, Excel (.xlsx), JSON
- **저장 정보:**
  - IP 주소 (필수)
  - 호스트명 (필수)
  - 운영체제 타입 (Linux만, 필수)
  - 계정 정보 (사용자명, 필수)
  - 인증 방식 (암호 또는 SSH 개인키, 필수)
  - 포트번호 (기본: SSH 22)

#### 3.0.2 파일 파싱 및 오류 처리

- 필수 필드 검증 (IP, 호스트명, OS 타입, 계정)
- 형식 오류가 있는 행은 무시하고 나머지 진행
- 오류 행에 대한 상세 로그 기록
- 사용자에게 파싱 결과 보고 (정상 N개, 실패 M개)

#### 3.0.3 자격증명 보안 저장

- **암호화:** AES-256 암호화
- **마스터 키:** 환경변수 (INFRA_SECURITY_MASTER_KEY)에서 로드
- **저장 위치:** PostgreSQL의 암호화 필드
- **접근:** Admin만 암호화된 값에 접근, 복호화는 스캔 시에만 수행

#### 3.0.4 자산 관리 (기본)

- 자산 추가/삭제/수정/조회
- 자산별 최근 스캔 일시 및 결과 표시
- 자산 상태 (정상/경고/위험) 시각화

### 3.1 OS 정보 자동 추출 ✅ (포함 - Ansible 사용)

#### 3.1.1 대상 플랫폼

- **Linux:** Ansible을 통한 SSH 기반 (기본 22번 포트)
- **Windows:** ❌ MVP 범위 제외 (다음 단계)

#### 3.1.2 접근 및 재시도 정책

- **병렬 처리:** 동시 스캔 서버 수 설정 가능 (기본값: 5~10개)
- **재시도 정책:** 접근 실패 시 3회 재시도, 30초 간격
- **실패 처리:** 실패한 서버는 건너뛰고 나머지 스캔 계속 진행
- **타임아웃:** 서버당 최대 5분

#### 3.1.3 추출 정보

- **기본 정보:**

  - OS 버전, 커널 버전
  - CPU, 메모리, 디스크 정보
  - 호스트명, IP 정보
- **패키지 정보:**

  - 설치된 패키지/소프트웨어 목록 및 버전
  - 주요 서비스 실행 상태
- **보안 설정값:**

  - 방화벽 상태 및 규칙
  - SSH 설정값
  - 사용자 권한 설정
  - SELinux 상태

#### 3.1.4 민감 정보 처리

- **필터링 대상:** 비밀번호, API 키, 프라이빗 키, 토큰 등
- **저장 방식:** 마스킹 처리 (****로 표시) 후 저장

#### 3.1.5 추출 방식

- **Linux:** Ansible을 통한 SSH 스크립트 기반 원격 실행
- **학습 전략:** 1시간 기초 학습 + Claude Code로 자동 생성

---

### 3.2 AI 기반 취약점 분석 ✅ (포함)

#### 3.2.1 분석 기준

- **CVSS 스코어 기반:** CVE 데이터에 포함된 CVSS v3.1 스코어 활용
- **설정값 기반:** 추출된 OS 설정과 보안 정책 비교

#### 3.2.2 심각도 분류

- **Critical (CVSS 9.0~10.0):** 즉시 조치 필요
- **High (CVSS 7.0~8.9):** 긴급 조치 필요
- **Medium (CVSS 4.0~6.9):** 계획된 조치
- **Low (CVSS 0.1~3.9):** 모니터링

#### 3.2.3 AI 분석 결과 형식

```json
{
  "vulnerability_id": "CVE-2026-XXXX",
  "title": "OpenSSL Buffer Overflow",
  "severity": "Critical",
  "cvss_score": 9.8,
  "affected_servers": [
    {
      "server_name": "web-01",
      "affected_package": "openssl-1.1.1a",
      "remediation_priority": 1
    }
  ],
  "root_cause": "OpenSSL 1.1.1 버전의 메모리 할당 오류로 인한 버퍼 오버플로우",
  "impact": "원격 코드 실행 (RCE) 가능",
  "remediation": "OpenSSL를 1.1.1w 이상으로 업그레이드",
  "references": ["https://nvd.nist.gov/vuln/detail/CVE-2026-XXXX"]
}
```

#### 3.2.4 AI 언어 설정

- **기술 용어:** 영어 (CVE, 패키지명, 설정값)
- **설명문:** 한국어 (원인, 영향도, 조치방법)

---

### 3.3 취약점 목록 웹 대시보드 ✅ (포함 - 기본 버전)

#### 3.3.1 메인 대시보드 (기본형)

- **KPI 메트릭:**

  - 전체 서버 수 / 정상 / 경고 / 위험 (기본 카드)
  - Critical / High / Medium / Low 취약점 수 (심각도 카드)
- **간단한 차트:**

  - 심각도별 취약점 분포 (파이 차트)

#### 3.3.2 서버 목록 조회

- 서버명, IP, OS, 마지막 스캔 시간
- 서버 상태 (정상/경고/위험)
- 취약점 개수 (심각도별)

#### 3.3.3 취약점 상세 조회

- CVE ID, 제목, 심각도, CVSS 스코어
- 영향받는 서버 목록
- 원인, 영향도, 권장 조치사항
- 참고 자료 (CVE 링크 등)

#### 3.3.4 데이터 갱신 주기

- 웹 대시보드는 1분마다 자동 갱신 (또는 수동 갱신)

#### 3.3.5 접근 권한 (간소화)

- **Admin:** 모든 서버 조회 + 설정 변경
- **Viewer:** 할당된 팀/부서의 서버 읽기만

---

### 3.4 CVE 데이터베이스 연동 ✅ (포함)

#### 3.4.1 실시간 CVE 모니터링 (기본)

- **외부 CVE DB:** NVD API, CVE-Details API
- **업데이트 주기:** 수동 또는 매일 자동 업데이트
- **실패 처리:**
  - API 다운 시 마지막 캐시된 CVE DB 사용

#### 3.4.2 CVE 매칭

- 추출된 패키지 버전과 CVE 데이터베이스 자동 매칭
- 조직 내 서버에 영향을 미치는 CVE 식별

## 5. 성공 기준 (MVP - 3개만) 🎯

### 필수 성공 기준 (이 3가지만 데모에서 보여주면 성공)

1. **자산 관리 + OS 정보 자동 수집**

   - ✅ CSV에서 서버 정보 입력 (IP, 호스트명, 계정)
   - ✅ Ansible/SSH로 Linux 서버 5~10대에서 OS 정보 자동 수집
   - ✅ 수집된 정보: 패키지 목록, OS 버전, 보안 설정
   - ✅ 수집 성공률 80% 이상
2. **AI 기반 취약점 분석**

   - ✅ Claude API가 수집된 정보를 분석하여 취약점 도출
   - ✅ 각 취약점에 심각도(Critical/High/Medium/Low) 할당
   - ✅ 조치 방안 제시 (한국어)
   - ✅ CVE 데이터와 매칭되어 영향받는 서버 식별
3. **웹 대시보드에 취약점 표시**

   - ✅ 웹 대시보드에서 취약점 목록 조회 가능
   - ✅ 심각도별 필터링
   - ✅ 서버별 취약점 분포 시각화
   - ✅ 각 취약점의 조치 방안 확인 가능

## 10. 가정 & 제약

- **테스트 환경:** Linux 서버 5~10대 (VM 또는 실제 서버)
- **CVE 데이터:** 공개 CVE DB 사용 (NVD API)
- **AI 분석:** Claude API 활용 (비용 추정: ~$5/20일)
- **성능:** 초기 버전은 수십 대 서버 지원
- **보안:**
  - 점검 결과는 조직 내부용
  - 자격증명은 AES-256 암호화
  - 민감정보는 마스킹 처리
- **네트워크:** SSH(22) 포트 통신 필수
- **팀 역량:** 개발 초심자 3명 + Claude Code 활용
