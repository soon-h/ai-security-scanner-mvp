# PRD: AI 기반 컨테이너 보안 점검 파이프라인 (MVP)

**작성일:** 2026-07-01
**기간:** 3주 (~7/15), 본인 단독 개발, 하루 평균 3~5h
**원본 스펙:** `spec_container_security_v2.md`

---

## Problem Statement

보안 담당자는 개발팀이 올린 여러 레포/컨테이너의 보안 설정을 **손으로** 점검하고 있다. 그래서:

- 점검 항목이 사람마다·시점마다 달라 **결과에 일관성이 없다.**
- Dockerfile 하드닝과 실행 컨테이너의 OS baseline을 **따로따로** 확인해야 해서 느리고 누락이 생긴다.
- 점검이 끝나도 raw 로그를 **사람이 읽을 수 있는 취약점 설명·심각도·개선안으로 정리**하는 데 시간이 많이 든다.
- 혼자서 여러 레포를 관리하다 보니 어느 레포가 어떤 상태인지 **한눈에 비교**할 방법이 없다.

## Solution

레포 URL 하나만 넣으면 **clone → Docker build → Sandbox 격리 실행 → Ansible 보안 점검 → Claude 분석 → Web Dashboard**까지 자동으로 한 바퀴 도는 파이프라인을 만든다.

- **일관된 점검 기준**(컨테이너 하드닝 + KISA 기반 OS/웹 점검 항목)을 Ansible playbook으로 고정한다.
- Claude가 raw 점검 결과(JSON)를 **취약점 설명·심각도·구체적 개선안**으로 변환한다.
- 여러 레포의 점검 이력·결과를 **대시보드에서 목록·상세로** 조회·비교한다.
- 사용자 경험: "레포 넣고 → 커피 한 잔 → 정리된 리포트 확인."

## User Stories

1. 보안 담당자로서, 대시보드에 GitHub 레포 URL을 입력하고 "점검 시작"을 누를 수 있기를 원한다. 그래야 수동 절차 없이 점검을 시작할 수 있다.
2. 보안 담당자로서, 입력한 레포가 자동으로 clone 되기를 원한다. 그래야 소스를 직접 내려받지 않아도 된다.
3. 보안 담당자로서, 레포의 Dockerfile이 자동 감지되고 `docker build`가 실행되기를 원한다. 그래야 이미지 빌드를 수동으로 하지 않아도 된다.
4. 보안 담당자로서, Dockerfile이 없는 레포는 명확한 사유와 함께 처리가 중단되기를 원한다. 그래야 무엇이 문제인지 바로 알 수 있다.
5. 보안 담당자로서, 빌드된 이미지가 네트워크 차단·읽기전용 FS·cap-drop·메모리/CPU 제한·타임아웃 하에 격리 실행되기를 원한다. 그래야 점검 대상이 내 환경을 위협하지 않는다.
6. 보안 담당자로서, 격리된 컨테이너에 점검 도구가 SSH 없이 붙을 수 있기를 원한다. 그래야 대상 컨테이너를 변형하지 않고 점검할 수 있다.
7. 보안 담당자로서, 실행 컨테이너와 Dockerfile에 대해 컨테이너/이미지 하드닝 항목(root 실행, 하드코딩 시크릿, 노출 포트 등)이 점검되기를 원한다. 그래야 일관된 하드닝 기준을 적용할 수 있다.
8. 보안 담당자로서, 실행 컨테이너에 대해 **KISA 주요정보통신기반시설 Unix 서버 점검 항목**(계정·파일권한·서비스·패치·로그)이 점검되기를 원한다. 그래야 표준 기준으로 OS 수준 취약 설정을 잡을 수 있다.
9. 보안 담당자로서, 점검이 시스템을 변경하지 않는 **읽기 전용**이기를 원한다. 그래야 점검 자체가 안전하다.
10. 보안 담당자로서, 각 점검 항목이 `{id, category, severity, status(pass/fail/skip), evidence}` 구조로 산출되기를 원한다. 그래야 결과를 기계적으로 다루고 Claude에 그대로 넘길 수 있다.
11. 보안 담당자로서, 대상 컨테이너에 해당 파일·서비스·데몬이 없어 점검이 불가능한 항목은 오류가 아니라 `skip`(사유 포함)으로 처리되기를 원한다. 그래야 최소화된 컨테이너라도 파이프라인이 중단되지 않는다.
12. 보안 담당자로서, fail로 판정된 각 항목에 대해 Claude가 취약점 설명·심각도·구체적 개선안을 생성하기를 원한다. 그래야 raw 로그를 직접 해석하지 않아도 된다.
13. 보안 담당자로서, Claude 출력이 고정된 JSON 스키마(`{id, severity, title, why_risky, remediation}`)로 나오기를 원한다. 그래야 파싱 실패·환각 없이 대시보드에 표시할 수 있다.
14. 보안 담당자로서, 점검이 진행되는 동안 단계별 상태(Clone → Build → Sandbox → Ansible → Claude → 완료)를 볼 수 있기를 원한다. 그래야 어디까지 진행됐는지 알 수 있다.
15. 보안 담당자로서, 별도 수동 개입 없이 결과 화면까지 수 분 내에 자동 도달하기를 원한다. 그래야 기다렸다가 결과만 확인하면 된다.
16. 보안 담당자로서, 레포별 점검 실행 이력을 목록으로 볼 수 있기를 원한다. 그래야 여러 레포의 점검 여부를 한눈에 파악할 수 있다.
17. 보안 담당자로서, 각 점검 실행의 심각도 요약(Critical/High/Med/Low 개수)을 볼 수 있기를 원한다. 그래야 위험도를 빠르게 가늠할 수 있다.
18. 보안 담당자로서, 항목별 Claude 리포트를 상세 화면에서 볼 수 있기를 원한다. 그래야 개별 취약점의 원인과 개선안을 확인할 수 있다.
19. 보안 담당자로서, 여러 레포의 점검 결과를 비교해 볼 수 있기를 원한다. 그래야 어느 레포가 더 위험한지 판단할 수 있다.
20. 보안 담당자로서, 점검 이력·결과가 로컬에 저장되기를 원한다. 그래야 대시보드를 다시 열어도 과거 결과를 볼 수 있다.
21. 보안 담당자로서, 빌드 실패·점검 오류 등이 발생하면 결과 화면에 원인이 표시되기를 원한다. 그래야 파이프라인이 조용히 죽지 않는다.
22. 보안 담당자로서, "취약 레포"와 "안전 레포"를 각각 점검했을 때 결과 대비가 뚜렷하게 드러나기를 원한다. 그래야 도구가 실제로 취약점을 잡는다는 것을 확인할 수 있다.
23. 보안 담당자로서, 컨테이너가 웹서버(Apache/IIS 등)를 포함하면 **KISA 웹서비스 점검 항목**도 함께 적용되기를 원한다. 그래야 웹 계층 취약 설정도 같은 파이프라인에서 잡을 수 있다.

## Implementation Decisions

- **파이프라인 오케스트레이터(단일 조율 지점):** Next.js API Routes(Node/TS)에 파이프라인 상태를 조율하는 오케스트레이터를 둔다. 레포 URL을 입력받아 clone → build → sandbox run → ansible → Claude → 저장 단계를 순차 실행하고, 단계별 상태를 관리한다. 동시 1개 레포 처리로 충분하다.
- **모듈 구성(논리적):**
  - *Repo/Build 모듈* — git clone, Dockerfile 감지, `docker build`. Dockerfile 부재 시 명확한 사유로 중단.
  - *Sandbox 모듈* — 제한된 `docker run`(`--network none`, `--read-only`, `--cap-drop ALL`, `--memory`, `--pids-limit`, 타임아웃).
  - *점검(Ansible) 모듈* — `ansible-playbook` 실행, 항목별 결과를 표준 스키마로 산출.
  - *분석(Claude) 모듈* — 점검 JSON을 입력받아 구조화 리포트 생성.
  - *저장 모듈* — 점검 이력·결과 영속화.
  - *Dashboard(Next.js/React)* — 입력·진행상태·이력·상세·비교 뷰.
- **Docker 실행 방식:** 호스트 Docker 소켓 마운트(`/var/run/docker.sock`)로 호스트 데몬을 사용해 build/run. DinD보다 단순하고 데모가 안정적. **로컬·단일 사용자 전제에서만** 사용하며 제품화 시 재검토(소켓 마운트 자체가 권한 리스크).
- **Ansible ↔ 컨테이너 연결:** `community.docker.docker` connection plugin 채택. SSH·에이전트 설치 없이 `docker exec` 기반으로 실행 컨테이너에 직접 붙어 점검.
- **점검 결과 스키마(계약):** 각 항목은 `{id, category, severity, status(pass/fail/skip), evidence}`. 이 JSON이 분석 모듈의 입력 계약이다.
- **점검 기준 채택:** OS/웹 점검 기준으로 **CIS 발췌 대신 KISA 「주요정보통신기반시설 기술적 취약점 분석·평가」 점검 항목**을 채택한다 — Unix 서버 67개(U-01~U-67) + 웹서비스 27개(W-01~W-27). 심각도는 가이드북의 상/중/하를 Critical/High/Med/Low 축으로 매핑한다(상→High 이상, 중→Med, 하→Low; C-02류 시크릿·빈 패스워드 등은 Critical로 상향 가능).
- **skip-safe 규칙:** 배포판을 사전 고정하지 않는다. Unix(U)·웹(W) 항목은 대상 컨테이너에 **해당 파일·서비스·데몬이 존재하지 않으면 `status: skip`**(evidence에 사유)으로 처리하고, 절대 파이프라인을 중단시키지 않는다. **최소화된 컨테이너 특성상 다수의 서비스·네트워크 항목(Finger·NFS·NIS·RPC·DNS·SNMP·FTP·메일·Telnet·웹 등)은 `skip`으로 집계될 수 있으며 이는 정상이다.** 컨테이너에 실제로 존재하는 축(계정 관리·파일/디렉터리 권한)이 fail/pass의 주된 대상이 된다.
- **Claude 출력 계약:** structured output(tool use)으로 `{id, severity, title, why_risky, remediation}` JSON 스키마를 강제. 모델은 `claude-opus-4-8`(비용상 대안 `claude-sonnet-5`).
- **저장소:** SQLite(또는 로컬 JSON/파일). 단일 사용자라 경량으로 충분.
- **점검 범위:** 컨테이너/이미지 하드닝 9개(C-01~C-09) + KISA Unix 67개(U-01~U-67) + KISA 웹서비스 27개(W-01~W-27), **총 103개 항목**을 목표로 한다. 단, 컨테이너에 미존재하는 항목은 skip-safe로 처리된다.
- **완주 안전장치(구현 순서):** 먼저 **얇은 슬라이스 3개(C-01·C-02·U-16)로 E2E를 끝까지 배선**한 뒤 항목을 확장한다. 확장 순서는 **컨테이너에 항상 존재하는 축(계정·파일권한: U-01~U-33) → 서비스/패치/로그(U-34~U-67) → 웹(W-01~W-27)** 순. 통합 리스크를 앞당기기 위함.

### 점검 항목

**① 컨테이너/이미지 하드닝** — Dockerfile 정적 분석(D) + 실행 컨테이너(R)

| ID | 항목 | 심각도 | 방법 | fail 기준 |
|----|------|:---:|:---:|------|
| C-01 | root(UID 0) 실행 | High | D+R | `USER` 미지정 or 실행 UID=0 |
| C-02 | 하드코딩 시크릿 | Critical | D | `ENV`/`ARG`에 password·token·key 패턴 |
| C-03 | 불필요한 노출 포트 | Med | D+R | 관리·DB 포트(22, 3306 등) `EXPOSE`/LISTEN |
| C-04 | base 이미지 태그 미고정 | Med | D | `:latest` 또는 태그 없음 |
| C-05 | 위험 패키지 잔존 | Med | R | curl/wget/gcc/apt 등 상주 |
| C-06 | setuid/setgid 바이너리 | High | R | 예상 외 setuid 바이너리 |
| C-07 | 쓰기 가능 루트 FS | Med | R | `--read-only` 미적용 시 쓰기 가능 |
| C-08 | HEALTHCHECK 부재 | Low | D | `HEALTHCHECK` 없음 |
| C-09 | ADD 원격 사용 | Low | D | `COPY` 대신 원격 URL `ADD` |

**② KISA Unix 서버 점검 (U-01~U-67)** — 실행 컨테이너(R), 미존재 시 skip-safe. 심각도는 가이드북 상/중/하 기준.

## Testing Decisions

- **좋은 테스트의 기준:** 구현 세부가 아니라 **외부로 관찰 가능한 행위**만 검증한다. 파이프라인 내부 함수 시그니처가 아니라, "레포 URL을 넣으면 어떤 결과 JSON/화면이 나오는가"를 본다.
- **최고 seam(단일 경계):** **Pipeline Orchestrator API**를 유일한 테스트 경계로 삼는다. 레포 URL을 입력하면 clone→build→sandbox→ansible→claude→저장이 한 바퀴 돌고 정규화된 결과가 나오는지를 이 한 지점에서 검증한다. 각 단계를 개별로 뚫지 않는다(seam 최소화).
- **테스트 대상 모듈:**
  - *오케스트레이터* — 준비된 "취약 레포" 픽스처를 입력하면 심어둔 취약 항목(예: C-01·C-02·U-16)이 fail로, "안전 레포"에서는 대부분 pass로 나오는지.
  - *점검 결과 정규화* — playbook 출력이 `{id, category, severity, status, evidence}` 스키마를 만족하고, **컨테이너에 미존재하는 항목(예: NFS·SNMP·웹)이 `skip`으로 처리되는지.**
  - *Claude 분석 계약* — fail 항목 입력 시 `{id, severity, title, why_risky, remediation}` 스키마로 파싱 가능한 출력이 나오는지(외부 호출은 필요 시 고정 응답으로 대체).
- **픽스처(=Prior art):** 성공 기준의 "취약 레포 1개 + 안전 레포 1개"를 그대로 테스트 픽스처로 재사용한다. 데모 시나리오와 테스트 대상이 일치하므로 유지보수 부담이 낮다.
- **비결정성 격리:** 실제 Docker 빌드·외부 Claude 호출은 느리고 비결정적이므로, 계약 검증 테스트에서는 이 경계를 고정 입출력으로 대체하고, E2E는 준비된 가벼운 레포로 1개만 돌린다.

## Out of Scope

- 실시간 CVE DB 연동 (별도 스펙 `spec_infra_security.md` 영역, 이번엔 정적/baseline 점검만)
- 자동 remediation(수정 적용) — Claude는 개선안 제안만, 실제 시스템 수정은 안 함
- 멀티유저 / 인증 / 권한 관리 — 본인 로컬 사용 전제, 로그인 없음
- CI/CD 웹훅 자동 트리거·스케줄링 — 점검은 대시보드에서 수동 실행
- 대규모 IDC 병렬 점검 — 동시 1개 레포 처리
- Kubernetes / 오케스트레이션 / 레지스트리 푸시 — 단일 Docker 호스트
- Dockerfile 없는 레포 자동 대응 — MVP는 Dockerfile 존재 레포만
- Windows 서버 점검(W-server) — KISA Web 중 IIS 항목은 포함하되 리눅스 컨테이너에서는 skip; Windows OS 자체 점검은 범위 밖

## Further Notes

- **최대 리스크(F3/Ansible):** 점검 항목이 103개(C 9 + U 67 + W 27)로 늘고 배포판 미고정 + 솔로 개발이 겹친다. → 완화: (1) 얇은 슬라이스 3개(C-01·C-02·U-16)로 E2E 먼저 관통, (2) **컨테이너에 항상 존재하는 축(계정 U-01~13, 파일권한 U-14~33)을 우선 구현**하고 서비스·웹 항목은 skip-safe로 흡수하며 후순위 확장, (3) 미존재 항목의 대량 `skip`은 정상 동작으로 간주.
- **컨테이너 적용성 주의:** KISA 항목은 완전한 서버 기준이라, 최소화된 컨테이너에서는 서비스·네트워크·웹 항목(Finger·NFS·NIS·DNS·SNMP·FTP·메일·Telnet·Apache/IIS)이 대부분 `skip`으로 나온다. 데모의 스포트라이트는 실제로 fail이 잡히는 계정·파일권한 축 + Claude 해석 품질에 둔다.
- **빌드 시간:** 무거운 이미지는 데모 시간 초과 가능 → 시연 레포는 가벼운 것으로 선정, 사전 1회 빌드로 캐시 워밍.
- **소켓 마운트 보안:** 로컬·본인 사용 전제를 벗어나면 반드시 재설계.
- **성공 기준(시연):** (A) 대시보드 입력→결과까지 무개입 E2E 자동 도달, (B) 심어둔 취약 설정 탐지 + 심각도·이유·개선안이 사람이 읽기 좋게 출력 + 안전 레포와의 대비.
- **버퍼 시 확장:** 진행 상태 실시간 스트리밍(SSE) → 심각도 대시보드 차트 → 웹(W) 항목 및 서비스(U-34~67) 커버리지 확대.
