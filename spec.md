# AI 기반 인프라(OS) 보안 점검 도구 - MVP Spec (고도화 버전)

**작성일:** 2026-06-16  
**팀:** 3명  
**기간:** 20일  
**목표:** AI 기반 보안 점검 도구로 IDC 규모의 인프라 보안 자동화

---

## 1. 문제 정의

**현재 상황:**
- 인프라(OS) 보안 점검을 수동으로 진행 중
- 매일 업데이트되는 CVE를 수동으로 모니터링
- 주요정보기반시설 취약점 가이드를 기준으로 한 점검이 비효율적

**해결하려는 것:**
- **자동화:** OS 정보를 자동으로 추출하고 AI가 분석
- **조기 발견:** 취약점을 빠르게 식별하고 보고
- **대규모 관리:** IDC 수준의 대량 서버를 효율적으로 점검
- **지속적 모니터링:** CVE 데이터베이스를 실시간으로 연동

---

## 2. 핵심 사용자 (페르소나)

**주요 사용자:** 보안팀 (Security Engineer, Sysadmin)

- 대규모 IDC 환경의 수백~수천 대 서버를 관리
- 보안 취약점을 빠르게 파악하고 대응해야 함
- 점검 결과와 이력을 추적하고 경영진에게 보고

**역할별 사용자:**
- **Admin:** 시스템 설정, 사용자 관리, 모든 서버 조회 및 관리
- **Operator:** 스캔 실행, 조치 상태 업데이트, 재스캔, 취약점 조회
- **Viewer:** 대시보드 조회, 리포트 다운로드 (읽기 전용)

---

## 3. 핵심 기능 (MVP)

### 3.0 자산 관리 및 인벤토리

#### 3.0.1 자산관리대장 (Asset Inventory) 파일 지원
- **파일 형식:** CSV, Excel (.xlsx), JSON
- **저장 정보:**
  - IP 주소 (필수)
  - 호스트명 (필수)
  - 운영체제 타입 (Linux/Windows, 필수)
  - 계정 정보 (사용자명, 필수)
  - 인증 방식 (암호 또는 SSH 개인키, 필수)
  - 포트번호 (기본: SSH 22, WinRM 5985)
  
#### 3.0.2 파일 파싱 및 오류 처리
- **파싱 전략:**
  - 필수 필드 검증 (IP, 호스트명, OS 타입, 계정)
  - 형식 오류가 있는 행은 무시하고 나머지 진행
  - 오류 행에 대한 상세 로그 기록
  - 사용자에게 파싱 결과 보고 (정상 N개, 실패 M개)

#### 3.0.3 자격증명 보안 저장
- **암호화:** AES-256 암호화
- **마스터 키:** 환경변수 (INFRA_SECURITY_MASTER_KEY)에서 로드
- **저장 위치:** PostgreSQL 또는 MySQL의 암호화 필드
- **접근:** Admin/Operator만 암호화된 값에 접근, 실제 복호화는 스캔 시에만 수행

#### 3.0.4 자산 관리
- 자산 추가/삭제/수정/조회
- 자산별 최근 스캔 일시 및 결과 표시
- 자산 상태 (정상/경고/위험) 시각화

### 3.1 OS 정보 자동 추출

#### 3.1.1 대상 플랫폼
- **Linux:** SSH 기반 (기본 22번 포트)
- **Windows:** WinRM 기반 (기본 5985번 포트)

#### 3.1.2 접근 및 재시도 정책
- **병렬 처리:** 동시 스캔 서버 수 설정 가능 (기본값: 10~20개)
- **재시도 정책:** 접근 실패 시 3회 재시도, 30초 간격
- **실패 처리:** 실패한 서버는 건너뛰고 나머지 스캔 계속 진행
- **타임아웃:** 서버당 최대 5분

#### 3.1.3 추출 정보
- **기본 정보:**
  - OS 버전, 커널 버전
  - CPU, 메모리, 디스크 정보
  - 호스트명, IP 정보

- **패키지 정보:**
  - 설치된 패키지/소프트웨어 목록 및 버전
  - 주요 서비스 실행 상태

- **보안 설정값:**
  - 방화벽 상태 및 규칙
  - SSH/WinRM 설정값
  - 사용자 권한 설정
  - SELinux/AppArmor 상태

- **시스템 로그 및 이벤트:**
  - 최근 로그인 기록
  - 보안 관련 이벤트

#### 3.1.4 민감 정보 처리
- **필터링 대상:** 비밀번호, API 키, 프라이빗 키, 토큰 등
- **저장 방식:** 마스킹 처리 (****로 표시) 후 저장
- **목적:** 감사 추적 (민감정보 검출 기록) + 보안

#### 3.1.5 추출 방식
- **Linux:** SSH 스크립트 기반 원격 실행 또는 Ansible
- **Windows:** PowerShell 원격 실행 (WinRM) 또는 Ansible

### 3.2 AI 기반 취약점 분석

#### 3.2.1 분석 기준
- **CVSS 스코어 기반:** CVE 데이터에 포함된 CVSS v3.1 스코어 활용
- **주요정보기반시설 보안 가이드:** KISA 가이드 기준 체크리스트
- **설정값 기반:** 추출된 OS 설정과 보안 정책 비교

#### 3.2.2 심각도 분류
- **Critical (CVSS 9.0~10.0):** 즉시 조치 필요
- **High (CVSS 7.0~8.9):** 긴급 조치 필요
- **Medium (CVSS 4.0~6.9):** 계획된 조치
- **Low (CVSS 0.1~3.9):** 모니터링

#### 3.2.3 AI 분석 결과 형식
```json
{
  "vulnerability_id": "CVE-2026-XXXX",
  "title": "OpenSSL Buffer Overflow",
  "severity": "Critical",
  "cvss_score": 9.8,
  "affected_servers": [
    {
      "server_name": "web-01",
      "affected_package": "openssl-1.1.1a",
      "remediation_priority": 1
    }
  ],
  "root_cause": "OpenSSL 1.1.1 버전의 메모리 할당 오류로 인한 버퍼 오버플로우",
  "impact": "원격 코드 실행 (RCE) 가능",
  "remediation": "OpenSSL를 1.1.1w 이상으로 업그레이드",
  "references": ["https://nvd.nist.gov/vuln/detail/CVE-2026-XXXX"]
}
```

#### 3.2.4 AI 언어 설정
- **기술 용어:** 영어 (CVE, 패키지명, 설정값)
- **설명문:** 한국어 (원인, 영향도, 조치방법)

### 3.3 취약점 목록 웹 대시보드

#### 3.3.1 메인 대시보드
- **KPI 메트릭:**
  - 전체 서버 수 / 정상 / 경고 / 위험 (기본 카드)
  - Critical / High / Medium / Low 취약점 수 (심각도 카드)
  - 미처리 / 진행 중 / 완료 조치 현황 (상태 카드)

- **차트:**
  - 취약점 추이 (시간별 그래프)
  - 서버별 취약점 분포 (막대 그래프)
  - 심각도별 비율 (파이 차트)

#### 3.3.2 서버 목록 조회
- 서버명, IP, OS, 마지막 스캔 시간
- 서버 상태 (정상/경고/위험)
- 취약점 개수 (심각도별)

#### 3.3.3 취약점 상세 조회
- CVE ID, 제목, 심각도, CVSS 스코어
- 영향받는 서버 목록 (간단 목록 + 상세 비교)
  - 각 서버의 해당 취약점 상태, 심각도, 조치 상태
- 원인, 영향도, 권장 조치사항
- 참고 자료 (CVE 링크 등)

#### 3.3.4 데이터 갱신 주기
- 웹 대시보드는 1분마다 자동 갱신
- 스캔 중일 때는 진행 상황 실시간 표시

#### 3.3.5 접근 권한
- **Admin:** 모든 서버 조회 + 설정 변경
- **Operator:** 할당된 팀/부서의 서버만 조회 + 조치 상태 업데이트
- **Viewer:** 할당된 팀/부서의 서버 읽기만

---

### 3.4 대량 일괄 점검 (Bulk Scan)

#### 3.4.1 멀티 서버 스캔
- **최대 규모:** 수천 대 서버 한 번에 점검 가능
- **병렬 처리:** 동시 스캔 수 설정 가능 (기본값: 10~20개)
- **진행 상황:** 실시간 모니터링 (완료/진행중/실패 서버 수)
- **예상 시간:** 스캔 진행률 및 예상 완료 시간 표시

#### 3.4.2 스캔 전략
- 실패한 서버는 건너뛰고 나머지 계속 진행
- 실패 서버는 자동으로 3회까지 재시도
- 스캔 완료 후 결과 요약 리포트 생성

#### 3.4.3 스케줄링
- **정기 스캔:** Admin이 시간 설정 가능
  - 예: "매일 오전 2시", "매주 월요일 오전 2시" 등
- **긴급 스캔:** 수동 트리거로 즉시 스캔 시작
- **스케줄 저장:** 설정된 스케줄은 DB에 저장

---

### 3.5 CVE 데이터베이스 연동

#### 3.5.1 실시간 CVE 모니터링
- **외부 CVE DB:** NVD API, CVE-Details API
- **업데이트 주기:** 매일 자동 업데이트
- **실패 처리:** 
  - API 다운 시 마지막 캐시된 CVE DB 사용
  - 배경에서 자동 재시도

#### 3.5.2 CVE 매칭
- 추출된 패키지 버전과 CVE 데이터베이스 자동 매칭
- 조직 내 서버에 영향을 미치는 CVE 식별

#### 3.5.3 CVE 알림
- **알림 방식:** 이메일
- **대상:** 보안팀 (Admin/Operator)
- **내용:** CVE ID, 영향받는 서버, 심각도, 조치 권장사항
- **배송 주기:** 새로운 CVE 발견 시 즉시

---

### 3.6 점검 결과 이력 관리

#### 3.6.1 히스토리 추적
- 모든 점검 결과 저장 (타임스탐프 포함)
- 시간 경과에 따른 취약점 추이 추적
- 과거 취약점과 현재 상태 비교

#### 3.6.2 데이터 보존 정책
- **최근 1년:** PostgreSQL/MySQL의 활성 DB에 저장
- **1년 이상:** 압축 아카이브 파일로 저장 (S3 또는 로컬 저장소)
- **아카이브 조회:** Admin이 수동으로 아카이브 데이터 다운로드 가능

#### 3.6.3 이력 비교
- 특정 기간의 서버별 취약점 변화 추적
- "7일 전 대비 변화", "30일 전 대비 변화" 등 비교 분석

---

### 3.7 자동 리포트 생성

#### 3.7.1 리포트 유형
- **경영진 요약 리포트:** 핵심 지표, 위험도, 권장사항
- **상세 취약점 리포트:** 모든 취약점, 영향받는 서버, 조치 방법
- **부서별 리포트:** 특정 부서/팀의 서버만 포함

#### 3.7.2 리포트 생성 주기
- **정기 리포트:** 주 1회 (예: 매주 월요일 오전 9시)
- **긴급 리포트:** 긴급 CVE 스캔 후 즉시 생성
- **수동 생성:** 필요 시 Admin/Operator가 수동으로 생성

#### 3.7.3 내보내기 형식
- **PDF:** 보기 전용, 경영진 보고용
- **Excel:** 데이터 분석 및 필터링 가능

#### 3.7.4 커스터마이징
- 부서/팀별 필터링
- 심각도별 필터링 (예: Critical/High만)
- 기간별 필터링
- 서버 선택적 포함/제외

---

### 3.8 조치 결과 추적

#### 3.8.1 조치 관리
- 발견된 취약점별 조치 기록
- 조치 상태 (미처리 → 진행 중 → 완료)
- 담당자 지정 (시스템 등록 사용자 중 선택)
- 예정일 및 실제 완료일 기록

#### 3.8.2 담당자 지정
- **방식:** 시스템 사용자 목록에서 선택
- **자동 알림:** 담당자 지정 시 이메일 발송

#### 3.8.3 효과 검증
- 조치 완료로 마크한 취약점에 대해 자동 재스캔 트리거
- **재스캔 범위:** 해당 서버 전체 (모든 정보 다시 수집)
- **결과 비교:** 재스캔 전후 취약점 개수 변화 표시
- **보고:** 조치 효과 (개선/악화/변화 없음)

#### 3.8.4 예외 처리 (Whitelist)
- **전사 규칙:** Admin이 정의한 조직 차원의 무시 정책
  - 예: "Red Hat 7은 지원 종료되어 있어서 패치 불가"
- **서버별 예외:** 특정 서버의 특정 취약점 무시
  - 예: "서버-A는 네트워크 격리 상태라 위험 낮음"
- **예외 기록:** 사유, 승인자, 예외 기간

## 4. 사용자 인증 및 권한 관리

### 4.1 웹 대시보드 로그인
- **방식:** ID/PW 기반 로그인
- **세션:** 토큰 기반 (JWT 또는 세션 쿠키)
- **보안:** HTTPS 필수, 비밀번호는 해시화 저장

### 4.2 역할 기반 접근 제어 (RBAC)

| 역할 | 대시보드 조회 | 스캔 실행 | 조치 관리 | 설정 변경 |
|------|---------|--------|---------|---------|
| **Admin** | ✓ (전체) | ✓ | ✓ | ✓ |
| **Operator** | ✓ (할당된 범위) | ✓ | ✓ | ✗ |
| **Viewer** | ✓ (할당된 범위) | ✗ | ✗ | ✗ |

### 4.3 데이터 접근 권한
- **Admin:** 모든 서버 및 취약점 조회
- **Operator/Viewer:** 할당된 팀/부서의 서버만 조회

## 6. 성공 기준 (시연 시나리오)

다음을 화면에서 보여줄 수 있으면 **성공:**

1. **서버 정보 추출:**
   - 테스트 Linux 서버 1대의 OS 정보, 패키지 목록, 설정값이 자동으로 수집됨

2. **AI 취약점 분석:**
   - 수집된 정보를 AI가 분석해서 취약점 목록을 자동 생성
   - 각 취약점에 심각도(Critical/High/Medium/Low)가 표시됨

3. **웹 대시보드:**
   - 메인 대시보드에 KPI 메트릭 표시 (기본형 + 심각도)
   - 취약점이 웹 화면에 시각적으로 표시됨
   - 취약점별 상세 정보 (원인, 조치방법, 영향받는 다른 서버) 조회 가능

4. **대량 스캔:**
   - 여러 대의 서버(테스트 5~10대)를 동시에 스캔
   - 병렬 처리 수(기본 10~20대) 설정 가능
   - 진행 상황이 실시간으로 표시됨
   - 실패한 서버는 자동 재시도 (3회, 30초 간격)

5. **정기 스캔:**
   - Admin이 정기 스캔 시간 설정 가능 (예: 매일 오전 2시)
   - 설정된 시간에 자동 실행 확인

6. **CVE 모니터링:**
   - 최신 CVE DB가 로드되고, 조직 서버에 영향을 미치는 CVE 목록 표시
   - 새로운 CVE 발견 시 이메일 알림 전송

7. **리포트 생성:**
   - 요약 리포트를 PDF로 다운로드 가능
   - 상세 리포트를 Excel로 다운로드 가능

8. **조치 추적:**
   - 특정 취약점을 담당자 지정하고 상태를 "진행 중"으로 변경
   - 재스캔 후 해당 서버의 취약점 개선 현황 확인
   - 조치 전후 비교 표시

9. **사용자 인증:**
   - 웹 대시보드 로그인 (ID/PW)
   - 역할(Admin/Operator/Viewer)별로 조회 권한 다름

10. **데이터 보존:**
    - 최근 1년 데이터는 DB에 저장
    - 1년 이상 데이터는 압축 아카이빙

## 11. 가정 & 제약

- **테스트 환경:** 최소 5~10대의 테스트 서버 필요 (Linux 5대 + Windows 2~3대)
- **CVE 데이터:** 공개 CVE DB 사용 (실시간 연동, API 할당량 고려)
- **AI 분석:** Claude API 또는 OpenAI GPT 활용 (비용 고려, 배치 처리)
- **성능:** 초기 버전은 최대 수백 대 서버 지원 (향후 확장)
- **보안:** 
  - 점검 결과는 조직 내부용 (외부 공개 불가)
  - 자격증명은 AES-256 암호화
  - 민감정보는 마스킹 처리
- **스캔 간격:** 정기(사용자 설정) + 긴급(수동) 지원
- **네트워크:** SSH(22), WinRM(5985) 포트 통신 필수
